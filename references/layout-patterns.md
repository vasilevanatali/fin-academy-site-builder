# Layout Patterns

Use this reference when a Fin Academy / Master CFO page needs stronger visual composition, richer section variety, or a less generic landing-page feel.

## Design Principle

Make the page useful before making it decorative. Each visual choice should help the visitor understand the offer, compare options, trust the author, or take the next action.

Avoid making the page a stack of similar cards. Use different layout mechanics for different jobs: hero, proof, program, artifacts, tariffs, FAQ, and final CTA should not all look like the same component with different text.

## Page-Level Composition

Choose 3-5 layout moments for a long page:

1. **Strong hero.** Show product category, audience, concrete job/result, CTA, and visual proof in the first viewport.
2. **Proof strip or context band.** Use a dense row of verified facts, logos, screenshots, or artifacts.
3. **Deep explanation section.** Use editorial layout, split columns, timeline, or steps.
4. **High-contrast anchor.** Use a dark or saturated section for tariffs, guarantee/limits, or final CTA.
5. **Interactive or artifact moment.** Show a preview of the workbook, dashboard, prompt pack, lesson player, table, checklist, or site screenshot.

For short event pages, use fewer moments but keep variation: hero, speaker/program, registration, FAQ/footer.

## Hero Patterns

Pick one based on available assets.

### Hero With Product Artifact

Use when there is a screenshot, lesson preview, workbook, prompt pack, dashboard, checklist, or product UI.

Layout:

- left or top: offer, audience, support copy, CTA, date/price;
- right or lower: large artifact preview;
- underneath: proof strip or "what you get" chips.

Rules:

- the artifact must be readable enough to signal the real product;
- avoid dark blurred mockups when the product needs inspection;
- use badges only for real facts.

### Hero With Expert Focus

Use for webinars, консультации, разборы, or expert-led products.

Layout:

- large portrait or event photo;
- compact credentials near the author, not buried later;
- CTA and event details visible without scrolling;
- proof/context band immediately after hero.

Rules:

- use real credentials only;
- avoid oversized biography in the first viewport.

### Hero With Outcome Map

Use when the product is abstract: AI workflow, finance process, consulting, outsourcing.

Layout:

- headline with target job;
- 3-5 outcome nodes or process blocks;
- one primary CTA;
- optional mini-preview of the resulting artifact.

Rules:

- nodes must be concrete: "собрать ДДС", "проверить договор", "подготовить лендинг", not generic "ускорить работу".

## Section Patterns

### Proof Strip

Use early, before long explanations.

Good content:

- verified numbers;
- author credentials;
- screenshots;
- client/student quotes;
- public examples;
- artifacts included in the product.

If proof is weak or missing, use a context strip instead: "что разберем", "какие инструменты", "для каких задач".

### Problem / Job Section

Do not write abstract pain. Tie the section to practical jobs:

- "финансовый директор хочет быстрее собрать отчет";
- "эксперт хочет запустить страницу без подрядчика";
- "команда хочет не забыть аналитику и UTM";
- "владелец хочет понять, где AI реально применим".

Useful layouts:

- two-column "как сейчас / как должно быть";
- checklist of jobs;
- process map;
- scenario tabs by audience.

### Program / Curriculum

Use the structure that matches the product.

- For lessons: timeline or lesson cards.
- For long programs: modules with nested lessons.
- For tool excursions: route map or "станции экскурсии".
- For consulting: phases, deliverables, decision points.

Rules:

- show concrete outputs per module;
- avoid vague verbs like "познакомимся", "узнаем", "разберем возможности" without an artifact or result;
- keep mobile as vertical accordions or cards.

### Artifact Preview

Use when the offer includes something tangible.

Artifacts:

- prompt pack;
- workbook;
- financial model;
- dashboard;
- checklist;
- lesson player;
- site screenshot;
- SOHO/CRM workflow;
- before/after page fragments.

Useful layouts:

- large preview with callouts;
- horizontal scroll only for visual galleries, not critical text;
- tabs: "модель", "чеклист", "промпты", "запись";
- before/after split for site improvements.

Rules:

- do not fake artifacts;
- if screenshots contain private data, blur or replace before publishing;
- use placeholders only when clearly labeled.

### Tariffs / Payment

Choose based on the number of options.

- One offer: single focused payment block with included items and payment note.
- Two offers: side-by-side cards on desktop, stacked cards on mobile.
- Three or more offers: cards with clear comparison, optional tabs on mobile.

Rules:

- price, access term, and CTA must be visible together;
- do not mark "популярный" or "выбор большинства" unless confirmed;
- use invoice/SBP/installment notes only when confirmed;
- avoid wide comparison tables on mobile.

### FAQ / Objections

Use accordions or simple stacked questions.

Question groups:

- level and prerequisites;
- access and recordings;
- payment and legal entity;
- support/feedback;
- difference from other products;
- what happens after payment;
- refund or policy only if confirmed.

## Layout Variety Menu

Use these patterns to avoid a flat stack of similar sections:

- asymmetric hero with artifact preview;
- dense proof strip;
- editorial two-column section;
- timeline / route map;
- tabs for audiences or artifacts;
- before/after comparison;
- sticky side summary on desktop;
- accordion program on mobile;
- dark anchor section for price or final CTA;
- screenshot gallery with callouts;
- compact FAQ;
- final CTA with explicit next step.

Do not use all patterns at once. Pick the few that fit the offer and assets.

## Interactive Blocks

Use interaction only when it improves understanding.

Good candidates:

- tariff toggle;
- program accordion;
- artifact tabs;
- before/after slider;
- FAQ accordion;
- lightweight calculator only when formulas and assumptions are provided;
- copyable prompt preview;
- video/lesson preview modal.

Avoid:

- autoplay carousels with critical content;
- hover-only explanations;
- decorative counters with invented values;
- heavy animation that delays reading.

## Visual Detail Rules

Use:

- consistent grid and max-width;
- local fonts;
- tabular numbers for prices/stats;
- icon + text only where icons clarify scanning;
- subtle borders/shadows, not heavy card piles;
- real screenshots and generated visuals tied to the product;
- visible focus states for keyboard users;
- responsive image sizes and aspect ratios.

Avoid:

- one-note purple/blue gradient pages;
- beige/brown-only palettes without contrast;
- nested cards;
- huge hero text inside a card;
- tiny gray text;
- negative letter spacing;
- long Russian labels forced into one line;
- horizontal scrolling tables;
- decorative orbs as the main idea;
- external fonts on production school pages unless explicitly allowed.

## Mobile Composition

Design 390px first.

Mobile rules:

- hero headline, support copy, CTA, and one visual signal must fit cleanly;
- sticky CTA starts after the first viewport and never covers footer/content;
- price cards stack and keep CTA close to price;
- tabs remain tappable; if labels are long, use stacked buttons or accordions;
- screenshots are either readable or openable;
- timeline becomes vertical;
- comparison tables become cards;
- button text wraps cleanly or uses shorter labels.

## When To Ask For Assets

Ask before designing if the page depends on:

- speaker photos;
- product screenshots;
- lesson previews;
- workbook/model/checklist files;
- brand colors/logos;
- payment links;
- analytics IDs;
- verified testimonials or numbers.

If assets are missing, build a structurally correct draft with labeled placeholders and report `данных нет` for missing facts.

## Final Design QA

Before handoff, check:

- first viewport has offer, audience, CTA, and visual signal;
- page uses at least 3 distinct section patterns on long pages;
- no repeated generic card layout dominates the whole page;
- mobile 390px has no horizontal overflow;
- sticky CTA works and does not cover content;
- tariff/payment block is easy to compare on mobile;
- all visual proof is real, public, or clearly placeholder;
- no invented stats, deadlines, testimonials, or urgency;
- analytics and UTM checks still pass after layout changes.
